## 2. Competition — where KYA actually wins

The agent-trust field splits into two camps, and KYA sits outside both:

- **Reputation registries / star ratings (ERC-8004 Reputation Registry, marketplace review
  scores).** These aggregate reviews — even when reviews are tied to x402 payment proofs,
  they score a *single listing* and are gameable by a ring. KYA already audits *who*
  reviewed whom (self-review + review-ring detection) rather than trusting the aggregate.
- **Delivery-proof services (e.g. ProofGate, a real OKX ASP in this hackathon).** These
  verify a *deliverable after the fact* — did the agent deliver what the task asked
  (`pass`/`needs_review`/`fail`). Useful, but post-transaction and per-task, not a
  pre-transaction verdict on the *counterparty*.

**The gap none of them can close:** every one of them evaluates an agent *in isolation*.
None can tell you that the "independent provider" you are about to pay is **1 of 99 shells
on one wallet with 19 sales between them** — because OKX's own `agent search` API never
returns `ownerAddress`. KYA reconstructs the operator graph the marketplace structurally
hides, and prices the *wallet* behind the listing.

### Sharpest defensible differentiation (lead with this)

> KYA is the only trust layer that prices the **operator, not the listing** — and it can,
> because it rebuilds the `ownerAddress → agents` graph that OKX's own search API hides.
> Everyone else scores one listing. KYA is the only one that sees the wallet holding 99 of
> them.

Critical note on the two headline claims:
- The **operator-cluster finding** is the moat: empirical, live-verifiable, and structurally
  invisible to every competitor. Lead with it.
- The **`max_safe_usd` priced ceiling** is a good *second* wedge ("a decision, not an
  opinion") but it is soft — today it is a conservative *floor* (sales × cheapest fee)
  unless the on-chain wash gate is keyed on, which it is not by default. Do not lead a
  critical judge with it; it depends on data KYA does not fully have yet. The cluster graph
  does not have that weakness.

## 3. Feature shipped — `cluster` as a first-class, machine-readable signal

**Problem it fixes:** the "1 wallet owns 99 agents" finding already fed the *score* (via the
`owner_fleet` / `owner_fleet_info` signals), but it only reached a caller as a **sentence
buried in the `signals` array**. A calling agent could not branch on it programmatically.

**What I added** (`oracle/engine.py`): every verdict's `evidence` now carries a structured
`cluster` object, surfaced on `/verify` (and inside the paid `/audit`, which returns full
evidence):

```json
"cluster": {
  "owner": "0x3256c679…",
  "fleet_size": 99,
  "fleet_sales": 19,
  "sales_per_agent": 0.19,
  "templated_names": 99,
  "penalized": true,
  "risk": "high"          // high = penalty fired | disclosed = big but real customers | low
}
```

Now a buyer agent can do `if verdict.evidence.cluster.penalized: require more settled
volume` — the finding is a **product capability, not a slide**.

**Safety / scope:**
- Read-only. It is derived entirely from the fleet dict already computed by
  `store.fleet_for()`. No new API calls, no network, no writes.
- Does **not** change scoring — the `owner_fleet` signals already moved the score; this is
  pure disclosure alongside them. All prior fleet tests still pass unchanged.
- Deliberately **out of the signed core** (`canonical_core()` still signs only
  agent_id/verdict/score/confidence/max_safe_usd), so existing signatures stay stable —
  verified: the digest is unchanged by this field.
- `None` when the wallet is unknown or runs only this one agent (never invents a cluster).
- **Did not touch** the x402 payment path, the facilitator, EIP-3009 settle, `/audit`
  charging, or `oracle/audit_paid.py`.

**Tests:** +4 in `tests/test_engine.py` (farm → risk `high`+penalized; real 32-agent
business → `disclosed`+not penalized; no fleet → absent; solo operator → absent). Full suite
**140 passed** (was 136).
